# gstack 完整使用指南

这是一份面向日常使用者的中文版完整指南，基于仓库内的 `gstack/README.md` 与 `gstack/docs/skills.md` 重组而成。

如果你只想快速上手，先看：
- [技能速查表](./03-skills-cheatsheet.md)
- [快速使用指南](./01-quickstart.md)

## 什么是 gstack

gstack 不是单一工具，而是一套把 AI 编程代理按交付阶段分工的工作流。

它的价值不在于 skill 多，而在于把原本一句“全都做一下”的模糊请求，拆成不同阶段的明确动作。

| 阶段 | 典型工作 |
| --- | --- |
| Think / Plan | 产品思考、方案评审、设计审查 |
| Build / Debug | 实现、调试定位 |
| Review / Test | 代码审查、浏览器 QA |
| Ship / Reflect | 发版上线、文档补齐、事后复盘 |

## 核心心智模型

把 gstack 当成一条流水线就够了：

`Think → Plan → Build → Review → Test → Ship → Reflect`

你在任何时刻，只需要先回答一个问题：`我现在处于哪一个阶段？`

## 最常用的主流程

| 场景 | 推荐流程 | 说明 |
| --- | --- | --- |
| 新功能或新项目 | `/office-hours` → `/autoplan` → 实现 → `/review` → `/qa` → `/ship` | 先想清需求，再过完整方案审查，最后完成审查、测试和发布。 |
| 修 bug | `/investigate` → 修复 → `/review` → `/qa` | 先查根因，再修复，然后补代码审查和回归验证。 |
| 上线和观察 | `/ship` → `/land-and-deploy` → `/canary` | 适合从发版推进到上线验证，再进入发布后观察窗口。 |

## 完整技能目录

下面按职责分组，不按文件夹分组。

### 一、产品与方案

| 技能 | 作用 | 适用场景 |
| --- | --- | --- |
| `/office-hours` | 在问题还没想透时使用，逼你把“我想做什么”翻译成“用户到底痛在哪里、最小切口是什么、真正的产品是什么”。 | 新产品想法<br>新功能方向不明<br>想重新定义一个需求 |
| `/plan-ceo-review` | 从产品和范围的角度重新审视方案，重点不是怎么实现，而是“这个方向本身对不对”“是不是还能更强”“是不是应该收缩范围”。 | 功能看起来能做，但不确定值不值得这么做<br>想在扩张范围和收缩范围之间做取舍 |
| `/plan-eng-review` | 把方案变得可落地，关注架构、数据流、边界条件、状态机、失败路径、测试矩阵和信任边界。 | 实现前的技术锁定<br>需要一个扎实的工程执行计划 |
| `/plan-design-review` | 在写代码前把交互、状态、信息层级、移动端和视觉风险补完整。 | 有界面的功能<br>你担心最后会落成“AI 味很重”的普通页面 |
| `/plan-devex-review` | 专门审查开发者体验，适合 API、SDK、CLI、文档平台、开发工具。 | 你做的是给开发者用的产品<br>你想提前发现 onboarding 阻力 |
| `/autoplan` | 自动串联多个 planning review，减少你一个个手动跑的成本。 | 想快速把一个方案过完整 review 流程<br>不确定自己到底缺 CEO、Design、Eng 还是 DX 审查 |

### 二、设计与体验

| 技能 | 作用 | 适用场景 |
| --- | --- | --- |
| `/design-consultation` | 从零建立设计系统，输出更稳定的视觉方向和 `DESIGN.md`。 | 新项目还没有设计系统<br>你希望 agent 后续生成的 UI 更稳定 |
| `/design-shotgun` | 快速出多套设计方向，让你基于视觉方案做选择，而不是只在文字层面来回描述。 | 想看多个方向<br>不确定页面该长什么样 |
| `/design-html` | 把设计稿、设计方向或描述落成真实页面实现。 | 设计方向已经确定<br>你准备进入页面落地阶段 |
| `/design-review` | 对已经实现的页面做视觉审计和修复。 | 页面已完成，但质感不够<br>想系统做 UI polish |
| `/devex-review` | 不是看计划，而是直接测试真实的开发者体验，比如文档、安装流程、CLI 帮助、Getting Started 路径。 | 开发者产品已经初步可用<br>你要验证“计划中的 DX”是否真的兑现 |

### 三、调试、审查与质量

| 技能 | 作用 | 适用场景 |
| --- | --- | --- |
| `/investigate` | 先定位根因，再讨论修复，避免用几轮拍脑袋修复把问题越弄越乱。 | 500 错误<br>某个流程突然坏了<br>你只知道现象，不知道根因 |
| `/review` | 偏代码审查，关注 CI 之外的风险、生产行为、回归、缺失测试和结构性问题。 | 提 PR 前<br>分支改动完成后 |
| `/codex` | 让另一个模型给第二意见，适合高风险改动或你想做交叉验证时使用。 | 需要独立复核<br>希望发现单一模型可能漏掉的问题 |
| `/cso` | 偏安全审计，覆盖 OWASP、STRIDE、权限、注入、密钥和供应链等方向。 | 安全审计<br>发布前安全检查<br>定期安全 review |
| `/health` | 做代码库总体健康检查，通常会围绕测试、类型、lint、死代码等维度。 | 想看代码质量状态<br>想做阶段性质量盘点 |

### 四、浏览器、QA 与线上验证

| 技能 | 作用 | 适用场景 |
| --- | --- | --- |
| `/browse` | 给 agent 一个真实浏览器能力，用来打开页面、点击、截图、提取信息和复现场景。 | 你需要真实页面证据<br>你想让 agent 看见页面，而不只是读代码 |
| `/open-gstack-browser` | 打开可视化浏览器，适合你想亲眼看 agent 如何操作页面。 | 需要带界面的浏览器协作<br>需要人工接管一小段流程 |
| `/setup-browser-cookies` | 把本机浏览器的 cookies 导入给 gstack 的浏览器会话。 | 要测试登录后的页面<br>需要 authenticated session |
| `/qa` | 执行完整的测试流程，通常会发现问题、尝试修复、再验证。 | 验收功能<br>回归测试<br>准备发布前整体检查 |
| `/qa-only` | 只给报告，不动代码。 | 你只想先盘点问题<br>你要把修复和测试拆成两个阶段 |
| `/benchmark` | 关注性能基线、Core Web Vitals 和资源大小变化。 | 对比前后性能<br>追踪页面速度变化 |
| `/canary` | 部署后持续盯线上，关注 console error、页面失败和性能回退。 | 刚上线后的观察窗口<br>对线上稳定性缺乏把握时 |

### 五、发布与文档

| 技能 | 作用 | 适用场景 |
| --- | --- | --- |
| `/ship` | 进入正式发布前流程，通常会做同步、检查、测试、准备 PR 等动作。 | 代码准备提交<br>想让发布前动作规范化 |
| `/setup-deploy` | 一次性配置部署平台、健康检查和上线命令。 | 第一次接入部署自动化 |
| `/land-and-deploy` | 在 PR 已通过的前提下，继续完成合并、部署和上线验证。 | 从“PR 已批准”推进到“线上已验证” |
| `/document-release` | 根据已经发生的代码变化补 README、架构说明、贡献说明和其他文档。 | 功能刚发出，文档容易过期<br>你不想每次手动补文档 |

### 六、协作、记忆与安全

| 技能 | 作用 | 适用场景 |
| --- | --- | --- |
| `/pair-agent` | 让多个 AI agent 围绕同一个浏览器或任务协作。 | 你想让别的 agent 一起测试页面<br>你在做多 agent 协作 |
| `/learn` | 查看和管理 gstack 的项目记忆，让经验能跨会话复用。 | 想看系统已学到哪些偏好或坑<br>想清理或导出记忆 |
| `/checkpoint` | 保存和恢复工作状态，适合中断、切换上下文或跨会话接力。 | 今天先停在这里<br>后面还要接着做 |
| `/careful` | 给危险操作加提示，适合共享环境和生产环境。 | 你担心误删、误推、误执行破坏性命令 |
| `/freeze` | 把编辑范围限制在一个目录，减少误改无关代码的风险。 | 定点调试<br>想控制 agent 的编辑边界 |
| `/guard` | 同时打开 careful 和 freeze，是更强的安全模式。 | 高风险改动<br>生产问题排查 |
| `/unfreeze` | 解除 freeze 的编辑限制。 | 你要扩大编辑范围 |
| `/gstack-upgrade` | 升级 gstack 自身。 | 技能缺失、版本落后、文档和行为不一致 |

## 推荐使用策略

| 策略 | 核心原则 | 具体做法 |
| --- | --- | --- |
| 默认按阶段走 | 不要把 gstack 当成几十个零散指令。 | 先判断自己处于哪个阶段，再选对应 skill，跑完后再进入下一阶段。 |
| 不确定时优先自动化 | 不确定 review 路径时，先用自动编排降低决策成本。 | 不知道该跑哪个 plan review，先跑 `/autoplan`；不知道只是“看代码”还是“真实验收”，通常两个都要跑：先 `/review`，再 `/qa`。 |
| 问题类请求先走调查 | 先查根因，再谈修复。 | 看到“报错”“500”“昨天还能用今天不行”“为什么这里坏了”，默认优先 `/investigate`，而不是直接改。 |
| 网页相关验证优先走浏览器 | 页面问题不要只靠静态读代码判断。 | 涉及页面行为、交互、截图、登录态、视觉问题时，优先考虑 `/browse`、`/qa`、`/design-review`。 |

## 几个典型组合

| 场景 | 推荐组合 | 适合什么情况 |
| --- | --- | --- |
| 产品从 0 到 1 | `/office-hours` → `/plan-ceo-review` → `/plan-eng-review` → `/plan-design-review` | 从需求定义一路推进到产品、工程和设计方案定稿。 |
| 快速把一个方案过全套审查 | `/autoplan` | 想一次性跑完整 planning review，而不是手动判断先后顺序。 |
| 纯后端或内部系统功能 | `/plan-eng-review` → 实现 → `/review` → `/ship` | 几乎没有前端设计压力，重点是工程落地、代码审查和发布流程。 |
| UI/前端功能 | `/plan-design-review` → 实现 → `/design-review` → `/qa` | 重点在交互、视觉和真实页面体验。 |
| API / SDK / CLI | `/plan-devex-review` → 实现 → `/devex-review` | 面向开发者的产品，重点是 onboarding、文档和调用体验。 |
| 高风险发布 | `/review` → `/cso` → `/qa` → `/ship` → `/land-and-deploy` → `/canary` | 适合安全、稳定性和发布风险都比较高的改动。 |

## 常见误区

| 误区 | 为什么不对 | 更好的做法 |
| --- | --- | --- |
| 把 `/review` 当成万能检查 | 它很重要，但不替代浏览器测试，也不替代上线验证。 | 把 `/review`、`/qa`、`/land-and-deploy`、`/canary` 看成不同阶段的检查。 |
| 一切都从 `/office-hours` 开始 | 只有需求模糊时才值得这样做。一个已经很清楚的小修复，不需要强行走产品讨论。 | 先判断任务类型，再决定是否需要产品层面的发散和重定义。 |
| 看到 bug 就让 agent 直接改 | 容易把“现象修复”误当成“根因修复”。 | 先 `/investigate`，再修。 |
| 把 gstack 当成命令记忆游戏 | 真正需要记住的是场景，不是命令表。 | 先识别阶段和目标，再选 skill。 |

## 建议的学习顺序

如果你要把 gstack 真正用顺手，推荐按下面顺序熟悉：

| 先熟悉什么 | 为什么先学它 |
| --- | --- |
| `/office-hours` | 建立“先想清楚问题”的习惯。 |
| `/autoplan` | 学会把 review 流程自动串起来。 |
| `/investigate` | 养成先查根因、再动手修的节奏。 |
| `/review` | 建立基本的代码风险意识。 |
| `/qa` | 学会把真实验收和代码审查区分开。 |
| `/ship` | 熟悉发布前的标准动作。 |
| `/canary` | 补上“发布后还要观察”的意识。 |
| `/design-review` | 在页面类需求里建立视觉和交互质量标准。 |
| `/cso` | 对高风险改动补上安全视角。 |
| 其他辅助 skill | 最后再按需要扩展到协作、记忆、部署等辅助能力。 |

## 故障排查

| 问题 | 先检查什么 | 处理方式 |
| --- | --- | --- |
| 技能没有生效或没被识别 | 项目里的 `AGENTS.md` 或相应 agent 配置里，是否声明了 gstack 路由规则。 | 先确认 skill 路由和 agent 配置是否真的启用。 |
| gstack skill 本身不可用 | 本地安装或初始化是否完成。 | 按仓库说明执行 `cd .opencode/skills/gstack && ./setup`。 |
| 浏览器相关 skill 工作异常 | 是否缺登录态、是否需要可视化窗口、是否版本过旧。 | 依次检查 `/setup-browser-cookies`、`/open-gstack-browser`、`/gstack-upgrade`。 |

## 结语

学 gstack 最有效的方法不是读完所有文档，而是连续跑几次完整流程。

你真正要记住的不是所有命令，而是这条节奏：

`想清楚 → 定计划 → 实现 → 审查 → 测试 → 发布 → 复盘`
