# ForgeFlow 工作记录

状态：草稿
日期：2026-04-19

## 总结

ForgeFlow 是一个以工作流为核心的 `opencode` 插件，带有学习闭环。

这个产品不是聊天壳，也不是自动代理壳。它的核心价值是：每个任务在执行前必须先留下记录，执行后必须再做反思，这样下一次运行才能更好。

插件对外应该尽量少暴露命令。默认只保留一个入口，内部通过工作流阶段处理计划、bug 记录、技术决策、执行和学习更新。

## 产品形态

ForgeFlow 分三层：

1. 插件外壳
   - 提供 `opencode` 入口
   - 接收用户命令
   - 输出人类可读状态
   - 保持足够薄

2. 工作流引擎
   - 对任务进行分类
   - 在执行前强制生成对应记录
   - 运行阶段状态机
   - 处理重试、暂停和校验

3. 学习核心
   - 读取执行历史
   - 提取有用信号
   - 更新默认值、模板和启发式规则
   - 让后续运行变好，但不抢主控权

## 设计原则

这个产品应该遵循“先记录，后执行”的工作流。

也就是说：

- 开发工作先写计划记录
- 修 bug 先写 bug 记录
- 技术讨论先写决策记录

记录不是可有可无的文档，而是工作流的门禁。它让流程更安全，也让后续学习成为可能。

## 工作流模型

推荐的流程是：

```text
接收任务
  -> 分类
  -> 记录
  -> 校验
  -> 执行
  -> 反思
  -> 学习
```

### 各步骤含义

- `接收任务`：捕捉用户意图
- `分类`：判断任务是计划、bug、方案，还是其他已支持类型
- `记录`：生成或补齐所需的结构化产物
- `校验`：不完整就拒绝继续
- `执行`：允许真正开始干活
- `反思`：记录结果、偏差和后续动作
- `学习`：根据运行历史更新未来默认行为

## 必需记录

### 计划记录

用于开发工作之前。

建议字段：

- 目标
- 范围
- 约束
- 风险
- 验收标准
- 不做什么

### Bug 记录

用于修 bug 之前。

建议字段：

- 现象
- 复现步骤
- 预期行为
- 实际行为
- 影响范围
- 疑似根因
- 验证方案

### 决策记录

用于技术讨论或架构选择之前。

建议字段：

- 问题描述
- 候选方案
- 取舍
- 选定方案
- 放弃方案
- 决策理由

## 命令面

对外命令面应该尽可能小。

推荐默认值：

- `/forgeflow`

其他所有动作都应该是内部工作流阶段或生成产物，而不是新的用户命令。

## TS 和 Python 分工

最清晰的分法是：

### TypeScript

TypeScript 负责插件边界和工作流控制。

职责：

- `opencode` 集成
- 命令处理
- 工作流编排
- 状态切换
- 日志与产物写入
- 最终面向用户的输出

### Python

Python 负责分析和学习逻辑。

职责：

- 信号提取
- 历史分析
- 图关系处理
- 策略更新
- 结构化 JSON 输出

### 边界规则

Python 不能变成第二个控制器。

插件必须保留一个运行状态和执行控制的唯一事实来源。Python 只能给出建议输出，不能直接改工作流引擎。

## 通信协议

TS 和 Python 只通过结构化数据通信。

推荐协议：

- 请求传入 JSON
- 响应返回 JSON
- 错误使用结构化字段 `code`、`message`、`retryable`、`details`

建议演进路径：

1. MVP：TS 通过 CLI 方式拉起 Python
2. 后续：如果需要，再升级成本地 HTTP 或轻量服务边界

## 为什么这样划分

这个结构比通用 agent 壳更符合产品意图。

核心承诺不是“模型替你思考”，而是“你的工作一定会留下记录，而这些记录会让下一次运行变得更好”。

这是一个以工作流为产品、学习为能力的系统，不是披着工作流外衣的 AI 演示。

## 首版最小范围

第一版只支持三类任务：

1. 计划
2. Bug
3. 方案

这三类足够覆盖：

- 开发前先计划
- 修复前先记录
- 选择技术路线前先讨论

这样已经足以验证工作流和学习闭环，不会过早把协议铺得太散。
