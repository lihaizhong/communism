# gstack 完整使用指南

这是一份面向日常使用者的中文版完整指南，基于仓库内的 `gstack/README.md` 与 `gstack/docs/skills.md` 重组而成。

如果你只想快速上手，先看：
- [技能速查表](./gstack-skills-cheatsheet.md)
- [快速使用指南](./gstack-quickstart.md)

## 什么是 gstack

gstack 可以理解为一套“给 AI 编程代理分工”的工作流。它不是单一工具，而是把不同阶段的工作拆给不同角色：

- 产品思考
- 方案评审
- 设计审查
- 调试定位
- 代码审查
- 浏览器 QA
- 发版上线
- 文档补齐
- 事后复盘

真正有用的不是 skill 数量，而是它把软件交付的各阶段分开处理，避免你每次都用一个模糊 prompt 让 agent “全都做一下”。

## 核心心智模型

把 gstack 当成一条流水线：

`Think → Plan → Build → Review → Test → Ship → Reflect`

你在任何时刻，只需要回答一个问题：

“我现在处于哪一个阶段？”

## 最常用的主流程

### 新功能或新项目

`/office-hours` → `/autoplan` → 实现 → `/review` → `/qa` → `/ship`

### 修 bug

`/investigate` → 修复 → `/review` → `/qa`

### 上线和观察

`/ship` → `/land-and-deploy` → `/canary`

## 完整技能目录

下面按职责分组，不按文件夹分组。

### 一、产品与方案

#### `/office-hours`

适合在问题还没想透时使用。它会逼你把“我想做什么”翻译成“用户到底痛在哪里、最小切口是什么、真正的产品是什么”。

适用场景：
- 新产品想法
- 新功能方向不明
- 想重新定义一个需求

#### `/plan-ceo-review`

从产品和范围的角度重新审视方案。重点不是怎么实现，而是“这个方向本身对不对”“是不是还能更强”“是不是应该收缩范围”。

适用场景：
- 功能看起来能做，但不确定值不值得这么做
- 想在扩张范围和收缩范围之间做取舍

#### `/plan-eng-review`

把方案变得可落地。关注架构、数据流、边界条件、状态机、失败路径、测试矩阵和信任边界。

适用场景：
- 实现前的技术锁定
- 需要一个扎实的工程执行计划

#### `/plan-design-review`

在写代码前把交互、状态、信息层级、移动端和视觉风险补完整。

适用场景：
- 有界面的功能
- 你担心最后会落成“AI 味很重”的普通页面

#### `/plan-devex-review`

专门审查开发者体验。适合 API、SDK、CLI、文档平台、开发工具。

适用场景：
- 你做的是给开发者用的产品
- 你想提前发现 onboarding 阻力

#### `/autoplan`

自动串联多个 planning review，减少你一个个手动跑的成本。

适用场景：
- 想快速把一个方案过完整 review 流程
- 不确定自己到底缺 CEO、Design、Eng 还是 DX 审查

### 二、设计与体验

#### `/design-consultation`

从零建立设计系统，输出更稳定的视觉方向和 `DESIGN.md`。

适用场景：
- 新项目还没有设计系统
- 你希望 agent 后续生成的 UI 更稳定

#### `/design-shotgun`

快速出多套设计方向，让你基于视觉方案做选择，而不是只在文字层面来回描述。

适用场景：
- 想看多个方向
- 不确定页面该长什么样

#### `/design-html`

把设计稿、设计方向或描述落成真实页面实现。

适用场景：
- 设计方向已经确定
- 你准备进入页面落地阶段

#### `/design-review`

对已经实现的页面做视觉审计和修复。

适用场景：
- 页面已完成，但质感不够
- 想系统做 UI polish

#### `/devex-review`

不是看计划，而是直接测试真实的开发者体验，比如文档、安装流程、CLI 帮助、Getting Started 路径。

适用场景：
- 开发者产品已经初步可用
- 你要验证“计划中的 DX”是否真的兑现

### 三、调试、审查与质量

#### `/investigate`

先定位根因，再讨论修复。它的价值在于避免你用几轮拍脑袋修复把问题越弄越乱。

适用场景：
- 500 错误
- 某个流程突然坏了
- 你只知道现象，不知道根因

#### `/review`

偏代码审查。关注 CI 之外的风险、生产行为、回归、缺失测试和结构性问题。

适用场景：
- 提 PR 前
- 分支改动完成后

#### `/codex`

让另一个模型给第二意见，适合高风险改动或你想做交叉验证时使用。

适用场景：
- 需要独立复核
- 希望发现单一模型可能漏掉的问题

#### `/cso`

偏安全审计，覆盖 OWASP、STRIDE、权限、注入、密钥和供应链等方向。

适用场景：
- 安全审计
- 发布前安全检查
- 定期安全 review

#### `/health`

做代码库总体健康检查。通常会围绕测试、类型、lint、死代码等维度。

适用场景：
- 想看代码质量状态
- 想做阶段性质量盘点

### 四、浏览器、QA 与线上验证

#### `/browse`

给 agent 一个真实浏览器能力，用来打开页面、点击、截图、提取信息和复现场景。

适用场景：
- 你需要真实页面证据
- 你想让 agent 看见页面，而不只是读代码

#### `/open-gstack-browser`

打开可视化浏览器，适合你想亲眼看 agent 如何操作页面。

适用场景：
- 需要带界面的浏览器协作
- 需要人工接管一小段流程

#### `/setup-browser-cookies`

把本机浏览器的 cookies 导入给 gstack 的浏览器会话。

适用场景：
- 要测试登录后的页面
- 需要 authenticated session

#### `/qa`

执行完整的测试流程，通常会发现问题、尝试修复、再验证。

适用场景：
- 验收功能
- 回归测试
- 准备发布前整体检查

#### `/qa-only`

只给报告，不动代码。

适用场景：
- 你只想先盘点问题
- 你要把修复和测试拆成两个阶段

#### `/benchmark`

关注性能基线、Core Web Vitals 和资源大小变化。

适用场景：
- 对比前后性能
- 追踪页面速度变化

#### `/canary`

部署后持续盯线上，关注 console error、页面失败和性能回退。

适用场景：
- 刚上线后的观察窗口
- 对线上稳定性缺乏把握时

### 五、发布与文档

#### `/ship`

进入正式发布前流程，通常会做同步、检查、测试、准备 PR 等动作。

适用场景：
- 代码准备提交
- 想让发布前动作规范化

#### `/setup-deploy`

一次性配置部署平台、健康检查和上线命令。

适用场景：
- 第一次接入部署自动化

#### `/land-and-deploy`

在 PR 已通过的前提下，继续完成合并、部署和上线验证。

适用场景：
- 从“PR 已批准”推进到“线上已验证”

#### `/document-release`

根据已经发生的代码变化补 README、架构说明、贡献说明和其他文档。

适用场景：
- 功能刚发出，文档容易过期
- 你不想每次手动补文档

### 六、协作、记忆与安全

#### `/pair-agent`

让多个 AI agent 围绕同一个浏览器或任务协作。

适用场景：
- 你想让别的 agent 一起测试页面
- 你在做多 agent 协作

#### `/learn`

查看和管理 gstack 的项目记忆，让经验能跨会话复用。

适用场景：
- 想看系统已学到哪些偏好或坑
- 想清理或导出记忆

#### `/checkpoint`

保存和恢复工作状态，适合中断、切换上下文或跨会话接力。

适用场景：
- 今天先停在这里
- 后面还要接着做

#### `/careful`

给危险操作加提示，适合共享环境和生产环境。

适用场景：
- 你担心误删、误推、误执行破坏性命令

#### `/freeze`

把编辑范围限制在一个目录，减少误改无关代码的风险。

适用场景：
- 定点调试
- 想控制 agent 的编辑边界

#### `/guard`

同时打开 careful 和 freeze，是更强的安全模式。

适用场景：
- 高风险改动
- 生产问题排查

#### `/unfreeze`

解除 freeze 的编辑限制。

适用场景：
- 你要扩大编辑范围

#### `/gstack-upgrade`

升级 gstack 自身。

适用场景：
- 技能缺失、版本落后、文档和行为不一致

## 推荐使用策略

### 策略 1：默认按阶段走

很多人一开始最容易犯的错，是把 gstack 当成几十个零散指令。更有效的方式是：

1. 先判断自己在哪个阶段
2. 再选对应的 skill
3. 跑完后再进入下一阶段

### 策略 2：不确定时优先自动化

如果你不知道应该跑哪个 plan review，先跑 `/autoplan`。

如果你不知道只是“看代码”还是“真实验收”，那通常两个都要跑：先 `/review`，再 `/qa`。

### 策略 3：问题类请求先走调查

看到“报错”“500”“昨天还能用今天不行”“为什么这里坏了”，默认优先 `/investigate`，而不是直接改。

### 策略 4：网页相关验证优先走浏览器

涉及页面行为、交互、截图、登录态、视觉问题时，优先考虑 `/browse`、`/qa`、`/design-review`，不要只停留在代码静态分析。

## 几个典型组合

### 产品从 0 到 1

`/office-hours` → `/plan-ceo-review` → `/plan-eng-review` → `/plan-design-review`

### 快速把一个方案过全套审查

`/autoplan`

### 纯后端或内部系统功能

`/plan-eng-review` → 实现 → `/review` → `/ship`

### UI/前端功能

`/plan-design-review` → 实现 → `/design-review` → `/qa`

### API / SDK / CLI

`/plan-devex-review` → 实现 → `/devex-review`

### 高风险发布

`/review` → `/cso` → `/qa` → `/ship` → `/land-and-deploy` → `/canary`

## 常见误区

### 误区 1：把 `/review` 当成万能检查

它很重要，但它不替代浏览器测试，也不替代上线验证。

### 误区 2：一切都从 `/office-hours` 开始

只有需求模糊时才值得这样做。一个已经很清楚的小修复，不需要强行走产品讨论。

### 误区 3：看到 bug 就让 agent 直接改

这通常会把“现象修复”误当成“根因修复”。先 `/investigate` 更稳。

### 误区 4：把 gstack 当成命令记忆游戏

真正需要记住的是场景，不是命令表。

## 建议的学习顺序

如果你要把 gstack 真正用顺手，推荐按下面顺序熟悉：

1. `/office-hours`
2. `/autoplan`
3. `/investigate`
4. `/review`
5. `/qa`
6. `/ship`
7. `/canary`
8. `/design-review`
9. `/cso`
10. 其他辅助 skill

## 故障排查

### 技能没有生效或没被识别

先检查项目里的 `AGENTS.md` 或相应 agent 配置是否声明了 gstack 路由规则。

### gstack skill 本身不可用

按仓库说明，可以在本地执行：

```bash
cd .opencode/skills/gstack && ./setup
```

### 浏览器相关 skill 工作异常

优先检查：
- 是否需要导入 cookies：`/setup-browser-cookies`
- 是否需要可视化浏览器：`/open-gstack-browser`
- 是否需要升级 gstack：`/gstack-upgrade`

## 结语

学 gstack 最有效的方法不是读完所有文档，而是连续跑几次完整流程。

当你开始习惯下面这个节奏时，skill 数量就不再是负担：

`想清楚 → 定计划 → 实现 → 审查 → 测试 → 发布 → 复盘`
